## #006 | 2026-03-28 | Git 동기화 범위 불일치

**문제:** `ai-employee-memory`만 GitHub에 연결. 실제 업무 데이터가 있는 `clawd-*` 폴더 6개는 .git 없음. 에이전트가 작업한 데이터(재고, 출고이력 등)가 GitHub에 동기화되지 않아 3대 맥 간 데이터 불일치, 백업 부재.
**원인:** 초기 설정 시 "기억(memory)"만 동기화 대상으로 설정. 업무 데이터는 고려 안 됨.
**해결:** `ai-employee-memory` 리포에 각 부서의 `data/`, `knowledge/` 폴더 추가. `sync.sh` 업데이트하여 `clawd-*` → `ai-employee-memory` 양방향 동기화 (30분 크론).
**교훈:** 온톨로지에서 "데이터가 관계를 따라 흐른다"고 설계했지만, 실제로는 데이터가 로컬에 갇혀 있었음. **"기억"과 "데이터"를 분리하지 말 것.**
**일반화:** ✅ 판매 패키지 설치 시 반드시 전 부서 data/ 폴더를 GitHub 동기화에 포함. 체크리스트 필수.

## #015 | 2026-03-30 | OpenClaw exec→Claude Code 호출 시 새 브라우저로 열림

**문제:** 텔레그램 "스마트스토어 주문 확인해줘" → OpenClaw가 exec로 Claude Code 호출 → 브라우저가 열리긴 하지만 **기존 로그인된 크롬이 아닌 새 브라우저로 열림**. 코난쌤 시연에서는 기존 크롬에 붙었음.
**핵심 차이:**
- Claude Code 직접 실행 (터미널) → `--autoConnect`로 **기존 크롬에 붙음** ✅ (탭 71개 감지 확인)
- OpenClaw exec로 Claude Code 호출 → **새 브라우저 인스턴스 생성** ❌

**영향:** 스마트스토어 크롤링 크론잡(9시) 실행되지 않음. 로그인 세션 없는 새 브라우저로 실행되어 로그인 화면에서 멈춤.
**임시해결:** 터미널에서 `claude --autoConnect` 직접 실행 → 기존 크롬 연결 확인됨
**근본해결:** OpenClaw exec 방식 대신 다른 방법 필요
**일반화:** ✅ 판매 패키지에서 브라우저 세션 연결 방식 표준화 필요

## 교훈

### 1. **온톨로지는 이론, 실제는 파일과 도구**
설계 단계에서는 "데이터가 관계를 따라 흐른다"고 했지만, 실제로는 에이전트가 `read()`, `edit()` 도구를 제대로 사용해야 데이터가 움직임.

### 2. **"했습니다" 는 거짓말의 시작**
에이전트가 "처리했습니다"라고 답하는 순간을 의심하라. 반드시 파일을 열어서 실제로 변경됐는지 확인해야 함.

### 3. **복잡한 것보다 단순한 것**
session_send 같은 복잡한 통신보다 직접 응답이 더 안정적. 마케팅 문구에 속지 말고 단순한 방식을 택하라.

### 4. **GitHub는 AI 직원의 인프라**
3대 맥에 AI 조직을 분산하려면 GitHub 동기화는 선택이 아니라 필수. 모든 데이터가 연결되어야 AI 조직이 작동함.

### 5. **템플릿은 반복업무의 해답**
매번 새로 만들지 말고 템플릿을 만들어라. 시간 절약과 품질 일관성을 모두 잡는다.
