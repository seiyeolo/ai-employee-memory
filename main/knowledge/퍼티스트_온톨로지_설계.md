# (주)퍼티스트 온톨로지 설계

> 랭그래프적 사고: 노드(엔티티) + 엣지(관계) + 조건 분기
> 원툴 벤치마킹 + OpenClaw 에이전트 기반

---

## 1. 엔티티(노드) 정의

### 핵심 엔티티
```
[제품] ──── 퍼티스트 골프/파크/일본형/미국형
[부품] ──── 센서, 실리콘, 스티커, 박스, 그립, 배터리
[거래처] ── 케이태우, 서진카페트, 해외그립업체
[고객] ──── 개인(스마트스토어), 기관(학교), 해외(Amazon)
[직원] ──── 대표, 총괄이사, 상주직원, 시간제 3명
[에이전트] ─ 세미에이전트, 생산관리부장, 물류부장, 영업마케팅부장, 콘텐츠부장, 재무인사부장
[창고] ──── 기존창고, 신규창고(4/1~)
[채널] ──── 스마트스토어, Amazon, 학교납품, 와디즈
[콘텐츠채널] ─ 블로그, BAND, Facebook, Instagram, YouTube, LinkedIn
```

### 활동 엔티티
```
[생산] ──── 조립, 테스트, 포장
[입고] ──── 부품입고, 부자재입고
[출고] ──── 택배발송, 기관납품, 해외발송
[주문] ──── 스마트스토어주문, Amazon주문, 직접주문
[AS] ────── AS접수, AS처리, AS완료
[매출] ──── 판매매출, 납품매출
[지출] ──── 원자재비, 인건비, 마케팅비, 물류비
[업무] ──── 일상업무, 프로젝트, 긴급
```

## 2. 관계(엣지) 정의

### 제품 관계
```
[제품] --구성됨--> [부품] (BOM)
  퍼티스트 골프 고급형 --구성됨--> 본체(고급), 매트, 박스(흰색), 매뉴얼
  
[제품] --보관됨--> [창고]
  퍼티스트 골프 고급형 --보관됨--> 신규창고

[제품] --판매됨--> [채널]
  퍼티스트 골프 고급형 --판매됨--> 스마트스토어, Amazon
```

### 생산 관계 (그래프 흐름)
```
[부품입고] --트리거--> [재고증가]
[생산요청] --필요--> [부품확인] --부족시--> [발주요청] --승인--> [거래처발주]
[생산완료] --트리거--> [완제품재고증가]
[출고요청] --필요--> [재고확인] --부족시--> [생산요청]
[출고완료] --트리거--> [재고감소] + [매출기록]
```

### 조직 관계
```
[파이오니어] --지시--> [세미에이전트(오케스트레이터)]
[세미에이전트] --위임--> [각 부서장]
[부서장] --보고--> [세미에이전트] (session_send)
[세미에이전트] --보고--> [파이오니어] (텔레그램)

[업무] --소속--> [부서]
[업무] --담당--> [에이전트]
[업무] --관련--> [업무] (하위/상위)
[업무] --연관--> [회의록/피드백]
```

### 재무 관계
```
[주문] --생성--> [매출]
[입고] --생성--> [매입/지출]
[급여지급] --생성--> [지출]
[매출] - [매출원가] = [매출총이익]
[매출총이익] - [판관비] = [영업이익]
```

## 3. 조건 분기 (랭그래프 노드)

### 주문 처리 그래프
```
START
  │
  ▼
[주문접수] ── 채널 판단
  │
  ├── 스마트스토어 → [영업마케팅부] 확인
  ├── Amazon → [영업마케팅부] 확인 + 환율 계산
  ├── 기관납품 → [영업마케팅부] 견적서 확인
  │
  ▼
[재고확인] ── 생산관리부에 확인
  │
  ├── 재고 있음 → [출고지시]
  ├── 재고 부족 → [생산요청] → 대기
  │
  ▼
[출고/포장] ── 물류부 처리
  │
  ├── 국내택배 → 일일발송집계 (15:30)
  ├── 해외발송 → 별도 물류 프로세스
  ├── 기관납품 → 납품체크리스트
  │
  ▼
[발송완료] → [매출기록] (재무인사부)
  │
  ▼
END
```

### AS 처리 그래프
```
START
  │
  ▼
[AS접수] ── Google Forms → Google Sheets 자동 기록
  │
  ▼
[증상분류] ── 영업마케팅부 판단
  │
  ├── 전원/배터리 (50%) → 교체/수리
  ├── 센서/거리측정 (13%) → 교정
  ├── 레이저 (8%) → 부품 교체
  ├── 화재위험 → 🚨 긴급 회수
  │
  ▼
[유상/무상 판단]
  │
  ├── 보증기간 내 → 무상
  ├── 보증기간 외 → 유상 (평균 39,000원)
  │
  ▼
[수리/교체] → [발송] → [완료기록]
  │
  ▼
END
```

### 안전재고 알림 그래프
```
[재고확인] (매일 or 입출고 시)
  │
  ▼
[안전재고 비교]
  │
  ├── 정상 → PASS
  ├── 안전재고 이하 → [생산관리부 → 오케스트레이터 긴급 보고]
  │                    │
  │                    ▼
  │              [발주 필요 판단]
  │                    │
  │                    ├── 자동 발주 (설정 시) → [거래처 발주]
  │                    └── 수동 승인 → [파이오니어 확인] → [발주]
  │
  ▼
END
```

## 4. 에이전트 스킬 설계 (온톨로지 기반)

### 생산관리부 스킬
```
skill: 재고확인
  입력: 제품명 또는 부품명
  처리: 재고마스터 조회 → 안전재고 비교
  출력: 현재고, 안전재고, 상태(정상/주의/긴급)
  트리거: 텔레그램 요청 또는 크론(매일 09:00)

skill: 입고기록  
  입력: "센서 2000개 입고" (텔레그램 자연어)
  처리: 파싱 → 부품마스터 매칭 → 재고 업데이트 → 기록
  출력: 입고 확인 + 현재고 표시
  연관: 매입기록 생성 (재무인사부에 전달)

skill: 생산기록
  입력: "고급형 30대 생산" (텔레그램 자연어)
  처리: 제품 파싱 → BOM 기반 부품 차감 → 완제품 증가
  출력: 생산 확인 + 부품 잔여 + 완제품 재고
  트리거: 부품 부족 시 발주 알림
```

### 물류부 스킬
```
skill: 출고처리
  입력: "택배 15건 발송" 또는 주문 데이터
  처리: 재고 차감 → 발송 기록 → 일일 집계
  출력: 발송 확인 + 잔여 재고
  연관: 매출기록 생성 (재무인사부에 전달)

skill: 납품관리
  입력: "금산초 퍼티스트 8대 납품"
  처리: 거래처 매칭 → 재고 차감 → 납품서 생성 → 매출 기록
  출력: 납품 완료 + 견적서/납품서 참조
```

### 영업마케팅부 스킬
```
skill: 스마트스토어모니터링
  입력: 없음 (크론 자동)
  처리: 브라우저 → 대시보드 확인 → 데이터 추출
  출력: 신규주문, 배송현황, CS현황, 리뷰
  트리거: 매일 09:00, 14:00

skill: AS관리
  입력: 없음 (크론) 또는 "AS 현황"
  처리: Google Sheets CSV → 분석
  출력: 총건수, 상태별, 미처리건, 고장유형 분석
```

### 재무인사부 스킬
```
skill: 매출집계
  입력: "이번 달 매출" 또는 크론
  처리: 스마트스토어 + Amazon + 납품 매출 합산
  출력: 채널별 매출, 총 매출, 전월 대비

skill: 손익보고
  입력: 크론 (월말)
  처리: 매출 - 매출원가 - 판관비 계산
  출력: 월간 손익 보고서

skill: 임금명세서
  입력: 크론 (매월 25일)
  처리: 근태 데이터 → 급여 계산 → 명세서 생성
  출력: 직원별 임금명세서 + 교부 알림
```

### 콘텐츠부 스킬
```
skill: 콘텐츠기획
  입력: "이번 주 콘텐츠 기획"
  처리: 트렌드 분석 + 제품KB → 기획안 생성
  출력: 채널별 콘텐츠 기획안

skill: 다국어콘텐츠 (글로벌)
  입력: 한국어 원본 콘텐츠
  처리: 영어/일본어/중국어 번역 + 현지화
  출력: 다국어 콘텐츠 세트
```

## 5. 글로벌 온톨로지 확장

### 시장별 엔티티 추가
```
[시장] ── 한국, 미국, 일본, 중국
[시장] --사용--> [채널] (시장별 판매 채널)
[시장] --사용--> [통화] (KRW, USD, JPY, CNY)
[시장] --적용--> [규격] (한국형, 미국형, 일본형)
[시장] --요구--> [언어] (한국어, 영어, 일본어, 중국어)
[시장] --준수--> [법규] (각국 제품안전법, 관세)
```

```
[시장조사] → [규격확인] → [제품현지화] → [채널확보]

                    ├── 대리점 → 계약 체결
                    └── 자체몰 → 웹사이트 구축
                                            │
                                            ▼
                                    [판매시작] → [CS체계] → [성과분석]
```

### 뷰/보드 매핑

| 원툴 보드 | 퍼티스트 구현 | 접근 방식 |
|---------|------------|---------|
| 👑 CEO보드 | KPI 대시보드 스킬 | 텔레그램 "주간 보고" |
| 🦄 마이 보드 | 텔레그램 개인 대화 | "오늘 할 일" |
| 🎆 스페셜 보드 | 매출관리 대시보드 | "매출 현황" |
| 🏠 회사 홈 | COMPANY.md + TEAM_CHARTER.md | GitHub |

### 핵심 차이 (원툴 vs 퍼티스트)

```
원툴 (노션):
  - 10개 팀 × 사람이 입력 = 인력 필요
  - 8개 DB × 수동 관리 = 시간 소모
  - 웹 접속 필요 = 컴퓨터 앞에 있어야

퍼티스트 (OpenClaw):
  - 6개 에이전트 × AI 자동 = 인력 불필요
  - 데이터 자동 연동 = 온톨로지 기반 흐름
  - 텔레그램 = 어디서든, 말로만

"원툴이 500억 회사의 뼈대를 팔았다면,
 퍼티스트는 그 뼈대에 AI 근육을 붙인 것이다."
```

## 8. 원툴 전체 DB 구조 확정 (완전 벤치마킹)

### 원툴 데이터베이스 10개 (4개 축)

```
학습/결과관리 (3개)          업무관리 (3개)
├── 🔬 고객학습자료 DB      ├── 💼 업무 DB
├── 🔙 피드백 DB            ├── 📃 회의록 DB
└── 💡 아이디어 DB          └── 📁 위키 DB

목표관리 (2개)              조직관리 (2개)

```

### 핵심 발견: 모든 팀보드는 같은 DB의 다른 뷰

```
💼 업무 DB (하나의 중앙 DB)
    ├── 영업팀 뷰: 팀="영업" 필터
    ├── 마케팅팀 뷰: 팀="마케팅" 필터
    ├── 인사팀 뷰: 팀="인사" 필터
    ├── ... (모든 팀이 같은 DB를 다른 필터로 봄)
    └── CEO보드 뷰: 전체 (필터 없음)

= 하나의 진실의 원천 (Single Source of Truth)
```

### 퍼티스트 OpenClaw에서의 구현

```
원툴: 1개 DB + 10개 뷰 (팀별 필터)
퍼티스트: 6개 에이전트 + 공유 데이터 (온톨로지 관계)

차이:
  원툴 — 같은 테이블을 다른 필터로 봄 (정적)
  퍼티스트 — 각 에이전트가 자기 소관 데이터를 소유하되
            session_send로 관계를 따라 데이터가 흐름 (동적)

예:
  원툴: 매출 보고서 = 영업팀 위키에 사람이 작성
  퍼티스트: 매출 보고서 = 재무인사부가 자동 생성 + 영업마케팅부 데이터 수신

이것이 온톨로지의 힘:
  데이터가 관계를 따라 자동으로 흐르고,
  각 노드(에이전트)가 자기 역할에 따라 처리한다.
```
